# 今日学习笔记：图片审核与 URL 上传能力落地

## 一、今日完成内容概览

今天主要完成了图片模块的两项关键能力建设：

1. **图片审核流程**：补齐审核状态、审核信息字段以及审核接口能力，完善图片内容管理闭环。
2. **URL 上传图片**：支持通过图片链接进行上传，和本地文件上传共用统一模板流程。

这两项能力让图片服务从“可上传”升级到“可治理 + 多入口上传”。

---

## 二、我做了哪些核心实现

### 1）数据库与实体层：增加审核相关字段

- 在图片表结构中补充了审核状态、审核人、审核时间、审核消息等字段。
- 在 `Picture` 实体中同步新增对应属性，保证数据库和业务模型一致。

**收获**：任何“状态机型业务”（如审核）都应该先把数据模型设计完整，再写接口逻辑。

### 2）审核状态枚举化，避免魔法值

- 新增 `PictureReviewStatusEnum`，统一管理待审核、通过、拒绝等状态。
- 业务逻辑通过枚举校验状态合法性，提升可读性和可维护性。

**收获**：当状态存在生命周期时，用枚举比硬编码数字更稳，后续扩展成本更低。

### 3）服务层新增审核方法，做状态流转控制

- 在 `PictureService` 中扩展审核能力。
- 在实现层中补充审核参数校验、图片存在性校验、状态更新逻辑。
- 处理“重复审核”或“非法状态流转”这类边界情况。

**收获**：审核不是“改个字段”，而是受规则约束的状态流转。

### 4）控制层开放审核接口

- 在 `PictureController` 新增审核相关接口，串联 DTO → Service → 持久化更新。
- 使用独立的 `PictureReviewRequest` 承载审核参数，职责清晰。

**收获**：请求模型要按场景拆分，不要一个 DTO 承载所有动作。

### 5）上传体系升级：新增 URL 上传实现类

- 新增 `UrlPictureUpload`，支持从网络地址拉取图片并上传。
- 与 `FilePictureUpload` 一起复用 `PictureUploadTemplate` 模板流程，保证上传主流程一致。
- `FileManager` / 控制层接口同步支持 URL 上传入口。

**收获**：模板方法模式非常适合“主流程固定、输入来源不同”的上传场景。

## 三、今天的关键设计思路

### 思路 1：先统一流程，再扩展入口

上传流程（校验 → 处理文件 → 上传对象存储 → 回填元数据）保持统一，
文件上传和 URL 上传只是“输入源”不同，避免两套逻辑逐渐分叉。

### 思路 2：审核能力要可追踪

审核不仅有“结果”，还要有“谁在什么时候给了什么结论”。
因此审核人、时间、消息字段不可少，这样后续才便于审计和问题追溯。

### 思路 3：把约束收敛在服务层

Controller 尽量轻，业务规则（状态合法性、重复审核限制等）集中在 Service，
后续无论是接口调用还是内部任务调用都能复用同一套规则。

---

## 四、踩坑与改进点

1. **审核状态更新的幂等性**：
   - 如果同一图片被重复提交相同审核请求，需要明确是拦截、忽略还是覆盖。
2. **URL 上传的可靠性**：
   - 需要进一步补充下载超时、重试、大小限制、MIME 校验。
3. **错误信息可读性**：
   - 后续可把审核与上传相关异常细分为更明确的业务错误码。

## 五、可复述的面试表达（1 分钟版）

今天我在项目中完成了图片审核与 URL 上传两项能力。审核方面，我先从数据模型入手补齐审核状态、审核人、审核时间和审核消息字段，并通过 `PictureReviewStatusEnum` 做状态枚举化，随后在 Service 层实现审核规则和状态流转控制，在 Controller 层提供独立审核接口。上传方面，我基于模板方法模式新增了 `UrlPictureUpload`，让 URL 上传和本地文件上传复用统一主流程，既提高了扩展性，也避免了重复代码。整体上，这次迭代让我把图片模块从“能上传”推进到“可审核、可追踪、可扩展”的状态。
